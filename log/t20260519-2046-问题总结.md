# 问题诊断

## 原子性的问题我认为存在至少以下要点
  1. 不是所有节点都适合原子性，有些节点就是会存在综合性内容。
     - 该内容被判定为原子性差
```json
    "node": {
      "id": "C_2",
      "type": "学科常识",
      "content": "柠檬酸循环是需氧生物体内普遍存在的一系列生化反应，是糖类、脂肪和蛋白质氧化的共同途径。"
    },
```
  2. 目前的机器评分机制只关心其逻辑意义上的原子性，而不会太关心是否保持了原子性对图整体的意义
     - 就比如第 1 点提出的图 
  3. 我们当初引入原子性是希望其能够避免这样的问题：模型错误地将两个节点合并，导致图出现逻辑结构模糊的问题
  4. 依据目前的表现来看，在合适的提示词下，我们之前预设的问题并没有严重地显现出来
  5. 对于原子性的评分标准，我们可能需要细化到这种程度：
     - 之前的原子性总体上更关心其语言逻辑上是否等同于一个原子事实
     - 虽然也会关心可拆分程度对逻辑的影响，但不是作为主要标准
     - 之后经过可能的修改，原子性更能会将“可拆分程度对图逻辑结构的影响”作为主要的判断标准
     - 也就是说，语言逻辑上的原子性作为次要标准，而图逻辑上的原子性对图结构错误的潜在影响会作为主要标准
  6. 还有一些情况是确实是判断模型自己判断出问题
     - 此处被判断为原子性差，显然是不对的
```json
    "node": {
    "id": "I_1",
    "type": "逻辑推导",
    "content": "喙形的变化表明环境也发生了变化。",
    "parents": [
        "C_1"
    ],
    "reasoning_logic": "根据图示现象推导环境变化的可能性。"
    },
```

## 关系部分
- llava内，关系部分，最明显的是 Reasoning_Type_Accuracy，19 条关系 badcase 全部涉及这个指标。很多 O 节点只是“最终答案复述”或“把上一步结果改写为最终句子”，但 type 写成“综合归纳”。按照当前评分标准，“综合归纳”更像多个并行分支汇聚后的总结，所以这些 O 节点被打很低。这可能不是单个样本问题，而是 dag.py 的生成规范和 relationship 评分标准之间存在不一致：O 节点到底能不能默认写“综合归纳”，现在评分器显然不太接受。

- 还有一小部分是真正的关系问题。比如 18_O、4_O 这种，父节点和当前节点几乎是同一句话，形成了 C -> O 的直接复述关系，评分器认为依赖或推理逻辑不合理。2_O 也有依赖完整性和准确性问题，说明最后结论的父节点没有覆盖原 CoT 里真正需要的前提。
  - 即，不属于"综合归纳"类

- nvlm内，关系问题依然最集中在 Reasoning_Type_Accuracy。18 条 relationship 里有 17 条涉及它。很多 O 节点被标成“综合归纳”，但实际只是“最终答案承接”或“结果陈述”。还有一些 SMILES 任务中，“识别结构”和“转成 SMILES”被标成逻辑推导，评分器也不太认可。这说明现在五类类型对某些任务不够贴合，尤其是“答案复述”“格式转写”“结构表征”这几类。

- 少量依赖问题是真问题。11_O、17_O、1_O、7_O 这类是父节点和当前节点几乎同一句，属于直接复述，不像一个推理关系。12_O 是最终答案的父节点不够完整，导致完整度和准确度都低。15_O 的父节点带有“基于题目语境推断”，和最终“答案是乙醇”的关系不够干净。

- 当前“综合归纳”的定义确实太窄，导致大量最终结论节点没有合适标签，只能硬塞进综合归纳，然后被评分器扣分。

- 如果把“综合归纳”直接扩大成兜底类，那么这个标签会变得很混。以后看到“综合归纳”，就不知道它到底是多分支汇总，还是答案承接，还是无法分类。

- 建议相关部分：
  - 注：综合归纳的原文为

        2.4.1.5 综合归纳: 汇聚多个并行的逻辑分支，通过总结或者升华，得出最终陈述。例如，由“中国土地辽阔”“中国物产丰富”归纳出“中国是个大国”

  
  - 扩大关于综合归纳的范围。我们目前的问题可能是，综合归纳的范围过窄，但是有很多问题没有办法使用更加具体的归纳标签，所以只能使用综合归纳作为标签。综合归纳不应该仅仅视为一种综合性的归纳，他应该包含一种更加广阔的判定，也就是说只要套不上其余集中判定，那么全都丢到综合归纳里面。
  - 或者，我们不称呼他为综合归纳，我们应该先单独分出几类具体的，然后把剩余的不可分类的统一统一打包成一个新的名称，新的名称是什么我没有想好
  - 结合两份报告有一个问题是，很多时候o节点会直接抄上面的某个i节点或者c节点。我目前猜测的原因有两个，
     - 可能是cot结尾会复述一遍答案，对于这种情况，可能需要修改prompt做单独的说明；
     - 模型可能很弱智，真的在cot里面是这么说的，这种情况下，可能就需要如上次建议所说的启用一个默认的归纳逻辑项。
  - "结果陈述"这个逻辑类型设置很不错，我认为这个可能可以加进去，而且考虑到很多模型可能没有明确的一次性确定答案的能力，可能会在最后强调一遍答案，或者是cot的设计本身就会强调最后的答案，因此这个结论承接可能是很有必要的
